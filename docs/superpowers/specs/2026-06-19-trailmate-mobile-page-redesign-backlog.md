日期：2026-06-19

依据：用户提供的三张 TrailMate 移动端参考图、现有 Android Compose 实现、生产级手机端产品体验设计规格、OpenSpec `trailmate-production-mobile-ux-system`

## 1. 设计 Brief 回放

TrailMate 是一款中文户外徒步 Android App。它应该像用户参考图一样轻盈、清晰、有户外质感：象牙白背景、深苔绿主色、路线/地图作为视觉主角、iOS 式底部抽屉和轻状态反馈。

本轮产品设计目标不是增加页面数量，而是把现有能力变成真正可用的移动端动线：

- 行前：导入 GPX，获得个性化路线评估，补齐装备和离线准备。
- 途中：真实 GPS 定位，轻导航，检查点推进，轨迹记录，偏航恢复和安全分享。
- 事后：轨迹复盘，沉淀为后续评估证据。

核心约束：

- 历史活动、基础档案、身高体重、装备库存都是 AI 和评估的后台证据，不在真实页面主视觉区裸展示。
- 每个页面只承担一个主任务，首屏最多一个主按钮。
- 不把诊断信息、AI 依据、地图技术状态放在首屏主层级。
- 不新增服务端页面或社区路线库，不做商城化装备推荐。

## 3. 全局动线

### 3.1 首次使用

1. 打开 App，进入注册/登录。
2. 完成基础档案采集或跳过。
3. 进入首页，看到导入路线和当前路线卡。
4. 导入路线后进入路线详情的“评估”。
5. 根据评估结果进入“装备”或“路线驾驶舱”。

成功标准：

- 用户不会在首页看到历史证据包或基础档案详情。
- 用户不会被迫理解 AI 如何算分，先获得行动建议。

### 3.2 行前准备

1. 首页或路线 Tab 导入 GPX。
2. 路线详情“评估”给出推荐/谨慎/暂不建议。
3. 主动作只指向一个下一步：检查装备或进入路线。
4. 装备缺失时进入路线装备清单。
5. 离线、GPS、装备状态在路线驾驶舱中以状态条呈现。

成功标准：

- 路线评估只在路线详情完整展示，首页只展示摘要。
- 装备页不出现商品化推荐。

### 3.3 现场徒步

1. 用户进入路线详情“路线”。
2. 首屏看到地图、当前位置、当前检查点、进度和主按钮。
3. GPS 未授权时主按钮为“启用定位”。
4. GPS 可用但未记录时主按钮为“开始徒步”。
5. 记录中可暂停、继续、结束并查看复盘。
6. 偏航时优先显示恢复建议和安全分享入口。

成功标准：

- 不滚动即可看到地图、状态条和主要操作。
- 技术诊断只能通过二级入口展开。

### 3.4 事后复盘

1. 结束记录后进入本次轨迹复盘。
2. 数据 Tab 展示最近活动、计划对比、趋势。
3. 复盘结果进入后续评估证据，但不作为首页主内容。

成功标准：

- 数据页回答“我走得怎么样”，不是设置中心。
- 历史 GPX 管理作为能力证据入口出现，不喧宾夺主。

## 4. 底部 Tab 页面合同

### 4.1 首页

主问题：今天要准备哪条路线？

首屏结构：

1. 顶部品牌、天气、位置、通知。
2. 问候标题：“准备走哪条线？”
3. 主入口：“导入 GPX 文件”。
4. 当前路线摘要卡，包含路线名、距离、爬升、预计用时、风险标签和下一步。
5. 快速开始三入口：路线评估、轻导航、装备清单。
6. 无真实健康数据前，删除“今日步数/热量/户外时长”样式的今日概览；改为“最近记录”空状态或隐藏。

保留：

- 导入入口。
- 当前路线摘要。
- 快速开始。
- 天气和位置。

移动：

- 历史活动入口移动到数据 Tab。
- 基础档案入口移动到我的 Tab。
- 数据控制入口移动到我的 Tab 的隐私与数据区。

删除或降级：

- 无真实数据来源的“今日概览”。
- 首页中展开的证据、档案、历史活动明细。

视觉要求：

- 当前路线卡是首屏视觉主角，参考图 1 的大图路线卡比例。
- 快速开始为 3 个小入口，不扩展成工具宫格。
- 首页不使用多层卡片嵌套。

验收：

- 首屏只有一个强主按钮。
- 用户 3 秒内能知道下一步是导入路线或进入当前路线。
- 首页没有身高体重、历史 GPX 详情或 AI 证据列表。

### 4.2 路线 Tab

主问题：我有哪些路线，当前准备到哪一步？

首屏结构：

1. 当前路线准备卡。
2. 导入路线入口。
3. 路线列表。
4. 仅当有解析任务时展示导入队列。
5. 空状态显示“导入 GPX 开始评估”。

保留：

- GPX 导入。
- 当前路线入口。
- 路线列表和导入状态。

移动：

- 详细评估移动到路线详情“评估”。
- 轨迹复盘移动到数据 Tab。

删除或降级：

- 与首页重复的完整当前路线卡。
- 与评估页重复的风险解释。

视觉要求：

- 列表要像路线库，而不是后台队列。
- 当前路线卡承担“继续准备”的入口。

验收：

- 路线 Tab 不展示个人档案、历史证据和完整评估。
- 导入队列无任务时不可占据首屏。

### 4.3 装备 Tab

主问题：这次路线缺什么，我已有装备是否可用？

首屏结构：

1. 页面标题“装备”。
2. 当前路线选择胶囊。
3. 内部分段：路线清单 / 我的装备 / 详情。
4. 默认打开“路线清单”。
5. AI 装备建议横幅，说明基于路线和天气，动作是“使用本地清单”或“重新评估”。
6. 必备装备列表，状态分为已匹配、需检查、缺失。
7. 添加已有装备使用底部抽屉。

保留：

- 路线清单。
- 我的装备库存。
- 装备详情。
- 品牌型号录入。

移动：

- 本路线装备摘要可在路线详情“装备”Tab 中出现。
- 完整库存维护只在底部装备 Tab。

删除或降级：

- 商品化或广告式推荐语。
- 过多 Beta 和模型能力展示。

视觉要求：

- 参考图 3：上方路线胶囊、分段控件、清单状态、底部抽屉。
- 装备图片要服务识别，不能变成商品橱窗。

验收：

- 缺失项能直接添加已有装备。
- 品牌型号只来自用户输入或本地库存。
- 用户不会误以为 App 在卖装备。

### 4.4 数据 Tab

主问题：我走过什么，能力是否变化？

首屏结构：

1. 最近轨迹复盘卡。
2. 计划 vs 实际摘要。
3. 历史活动列表。
4. 能力趋势入口。
5. 历史 GPX 管理入口。

保留：

- 轨迹回顾。
- 历史活动。
- 评估证据管理入口。

移动：

- 数据清除和隐私移动到我的 Tab。
- 当前路线导航移动到路线详情。

删除或降级：

- 无数据时的假趋势。
- 原始 AI 输入展示。

视觉要求：

- 数据页更像复盘中心，不像仪表盘堆数值。
- 无轨迹时用清晰空状态：“完成一次记录后会出现复盘”。

验收：

- 数据页不承担路线导入主流程。
- 历史活动作为结论和列表展示，不作为后台证据包展开。

### 4.5 我的 Tab

主问题：账号、权限、隐私和基础设置在哪里？

首屏结构：

1. 用户摘要和基础档案编辑入口。
2. 权限状态：定位、通知、地图授权。
3. 数据与隐私：导出、清除本地数据、删除账户。
4. 地图设置：高德 Key 状态、离线地图入口。
5. 关于与安全边界。

保留：

- 基础档案摘要。
- 权限和地图授权状态。
- 数据控制能力。

移动：

- 路线评估、路线清单、轨迹复盘分别回到路线、装备、数据。

删除或降级：

- 我的页中的路线内容。
- 长篇能力证据说明。

视觉要求：

- 设置中心要安静、清晰、可扫描。
- 风险或权限异常用小状态提示，不占据整页主视觉。

验收：

- 我的 Tab 中没有路线详情和装备主流程。
- 用户可以找到修改基础档案、定位权限和清除数据的入口。

## 5. 路线详情四 Tab 页面合同

### 5.1 评估

主问题：这条线对我是否合适？

首屏结构：

1. 路线名、距离、累计爬升。
2. 分段控件：评估 / 路线 / 计划 / 装备。
3. 评估结果卡：推荐、谨慎、暂不建议。
4. 预计用时区间。
5. 最多 3 个风险因素。
6. 主动作：检查装备或进入路线。
7. “查看依据”折叠入口。

保留：

- 个性化评估结论。
- 可信度。
- 关键风险。

隐藏到二级：

- 历史 GPX 证据。
- 基础档案细节。
- AI 原始解释和调试信息。

验收：

- 首屏不会出现超过 3 个风险标签。
- 技术诊断不在评估页出现。
- 安全文案是辅助语气，不承诺安全。

### 5.2 路线驾驶舱

主问题：我现在在哪里，下一步怎么走？

首屏结构：

1. 地图占据 58%-65% 屏高。
2. 地图上有路线、检查点、当前位置、已记录轨迹。
3. 右侧或底部浮层展示评估摘要和进度。
4. 底部当前检查点卡。
5. 状态条：GPS、记录、离线、装备。
6. 主动作：启用定位 / 开始徒步 / 暂停 / 继续 / 查看复盘。

保留：

- AMap 地图。
- GPS 定位。
- 轨迹记录。
- 偏航恢复。
- 安全分享。
- 检查点推进。

移动到折叠区：

- 现场详情。
- 地图图层解释。
- 高德上线检查。
- 位置可靠性长说明。

验收：

- 不滚动即可完成现场主要操作。
- 状态条每一项都可点击进入对应处理。
- GPS 不可用时有恢复入口，而不是只显示失败。

### 5.3 计划

主问题：什么时候补给、休息、核对风险？

首屏结构：

1. 计划摘要。
2. 检查点时间线。
3. 每项显示距离、预计到达、建议动作。
4. 回撤或风险提醒。

保留：

- 检查点计划。
- 补给和休息建议。
- 风险核对。

删除或降级：

- 命令式措辞。
- 超过 12 项的长列表。

验收：

- 文案使用“建议检查”“建议补给”。
- 离线状态下仍可查看计划。

### 5.4 装备

主问题：本路线装备是否就绪？

首屏结构：

1. 装备准备摘要。
2. AI 装备建议横幅。
3. 必备装备清单。
4. 缺失项添加入口。
5. 前往完整装备库入口。

保留：

- 本路线装备判断。
- 已有装备匹配。
- 添加已有装备。

移动：

- 完整库存列表移动到底部装备 Tab。

验收：

- 路线详情装备 Tab 只服务当前路线。
- 不出现品牌广告或购买诱导。

## 6. Onboarding 页面合同

### 6.1 注册 / 登录

主问题：如何创建私密账户并开始？

首屏结构：

1. TrailMate 标识。
2. 简短安全边界。
3. 邮箱、密码。
4. 主按钮“继续”。
5. 登录/注册切换。
6. 隐私和位置数据入口。

验收：

- 不讲完整产品功能。
- 邮箱、密码错误有就地反馈。
- 可进入基础档案。

### 6.2 基础档案

主问题：路线评估最低需要哪些个人上下文？

首屏结构：

1. 进度：“基础档案 1/2”。
2. 平时运动。
3. 户外经验。
4. 身体和背包。
5. 主按钮“保存并进入首页”。
6. 次按钮“暂时跳过”。

验收：

- 身高体重标明可选，只用于保守估算。
- 完成后进入首页，不停留在档案展示页。

## 8. 实施优先级建议

### P0：锁定信息架构

目的：先解决“乱”和“重叠”。

范围：

- 首页只保留今日路线准备。
- 路线 Tab 只保留路线库和当前路线。
- 我的 Tab 收敛为账号、权限、隐私、设置。

验收：

- 底部五个 Tab 不再互相重复主任务。

### P1：路线详情首屏重构

目的：让真实 GPS 和轨迹记录成为现场主体验。

范围：

- 评估页隐藏证据详情。
- 路线驾驶舱首屏固定地图、状态条和主动作。
- 诊断信息折叠。

验收：

- 现场页不滚动即可开始定位或记录。

### P2：装备页生产化

目的：让装备变成路线准备工具，不像商城页。

范围：

- 路线清单默认首屏。
- 添加已有装备底部抽屉。
- 完整库存作为二级管理。

验收：

- 缺失装备可以闭环到用户已有装备。

### P3：数据和复盘

目的：让数据页成为事后复盘中心。

范围：

- 最近记录。
- 计划 vs 实际。
- 历史活动和能力趋势入口。

验收：

- 无轨迹时有可执行空状态。

### P4：视觉统一和动效

目的：接近三张参考图的质感。

范围：

- 当前路线大卡。
- iOS 式底部抽屉。
- 轻量状态动画。
- 地图浮层和装备清单视觉细化。

验收：

- 首页、路线驾驶舱、装备页截图与参考图在层级、留白、色彩、按钮密度上保持一致。

## 9. 页面级验收矩阵

| 页面 | 主任务是否唯一 | 首屏主按钮 | 后台证据是否隐藏 | 关键状态是否可处理 |
| --- | --- | --- | --- | --- |
| 注册/登录 | 创建账户 | 继续 | 是 | 表单错误可处理 |
| 基础档案 | 采集最低上下文 | 保存并进入首页 | 是 | 可跳过 |
| 首页 | 开始路线准备 | 导入路线或继续当前路线 | 是 | 当前路线下一步可处理 |
| 路线 | 管理目标路线 | 导入路线 | 是 | 导入状态可处理 |
| 评估 | 判断是否合适 | 检查装备或进入路线 | 是 | 风险可查看依据 |
| 路线驾驶舱 | 现场轻导航 | 启用定位/开始徒步 | 是 | GPS、记录、离线、装备均可处理 |
| 计划 | 查看补给休息建议 | 无强制主按钮 | 是 | 检查点可离线查看 |

## 10. 代码实现前置条件

进入代码实现前需要满足：

- 用户确认本 backlog 的页面职责和动线。

- 每个改动阶段都有截图或 Compose 测试验证。

## 11. 设计自检

- 页面职责：每个底部 Tab 和路线详情 Tab 都有唯一主任务。
- 视觉方向：持续贴近三张参考图，不引入重暗色或营销页结构。
- 信息隐藏：历史证据、档案、AI 输入和诊断默认不展示在主页面。
- 交互闭环：导入、评估、装备、定位、记录、复盘都有下一步。
- 安全边界：所有建议保持辅助语气，不替代现场判断、路标和离线地图。
