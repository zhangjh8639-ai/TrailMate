# TrailMate 生产闭环 V2 设计规格

日期：2026-06-23
状态：方向 A 已确认，待实现计划
范围：Android 手机端主动线、全屏导航职责、服务端装备目录、登录注册成熟度、真机验收

## 1. 设计目标

TrailMate 下一阶段要从“能跑的功能原型”升级为“用户能理解、能信任、能在真机上完成核心路径的徒步工具”。本轮不追求更多页面，而是把现有能力整理成一条闭环：

1. 用户用微信或手机号建立账号。
2. 用户填写最少必要的运动、户外、身体和负重信息。
3. 用户导入目标 GPX。
4. TrailMate 给出路线评估、计划检查点和装备建议。
5. TrailMate 从服务端装备目录匹配本次路线需要的品牌装备。
6. 用户进入现场全屏导航，完成 GPS 定位和轨迹记录。

基础档案、历史证据和 AI 输入仍然是后台依据；主页面展示结论、状态和下一步行动。

## 2. 产品原则

### 2.1 一个页面只回答一个问题

- 首页：今天准备哪条路线？
- 路线 Tab：我的目标路线在哪里？
- 路线详情评估：这条线对我是否合适？
- 路线详情路线：路线结构和检查点是什么？
- 全屏导航：我现在在哪里，下一步怎么走？
- 计划：什么时候补给、休息和复核风险？
- 装备：这条路线需要什么，服务端装备库有哪些候选能覆盖？
- 我的：账号、权限、隐私和基础档案在哪里？

### 2.2 服务端是可信数据源

装备品牌、型号、类别、重量、图片和适用场景来自服务端装备目录。用户不手填品牌型号创建装备；找不到型号时由后台补充目录后再让用户选择。

### 2.3 现场模式必须克制

全屏导航不展示高德上线检查、离线底图长说明、权限教育、装备解释或 AI 依据。它只展示现场需要的信息：地图、当前位置、路线、检查点、轨迹、GPS、记录和安全动作。

### 2.4 视觉继续贴近参考图

保留浅象牙白背景、深苔绿主色、地图大画面、轻量分段控件、圆角状态胶囊和 iOS 式底部抽屉。卡片要服务信息层级，不能把每个元素都包成卡片。

## 3. 用户动线

### 3.1 首次使用

1. 打开 App，进入账号页。
2. 默认展示微信登录，手机号登录作为次入口。
3. 登录成功后进入基础档案。
4. 基础档案完成或跳过后进入首页。

账号页只说明私密、安全边界和登录方式，不解释完整产品能力。

### 3.2 行前准备

1. 首页显示导入 GPX 主入口和当前路线评估卡。
2. 用户进入路线 Tab 管理目标路线。
3. 用户打开路线详情，先看到评估。
4. 评估给出匹配等级、预计用时、关键风险和下一步。
5. 若缺关键装备，引导去装备；若准备充分，引导去路线或全屏导航。

### 3.3 装备准备

1. 路线详情装备 Tab 显示本路线 AI 装备建议。
2. 缺失项按钮为“查看候选”或“选择候选”，不是“添加已有装备”。
3. 点击后打开服务端目录候选底部抽屉。
4. 用户查看类别、品牌、型号、缩略图、重量和适用标签。
5. 本路线清单从服务端目录直接更新匹配状态，不要求用户维护个人装备库。

### 3.4 现场徒步

1. 用户从路线详情进入全屏导航。
2. 全屏导航请求或复用真实定位。
3. 用户开始记录，轨迹点持续保存。
4. 偏航或 GPS 失准时显示恢复建议，但不自动推进检查点。
5. 结束记录后进入数据复盘。

## 4. 页面规格

### 4.1 登录注册

默认结构：

- TrailMate 标识和一句价值说明。
- 微信登录主按钮。
- 手机号登录折叠或次级入口。
- 登录中、失败、取消、后端未配置、微信未安装等状态。
- 隐私和安全边界说明。

成功标准：

- 用户能判断自己是在登录还是注册。
- 微信未配置时给出可理解提示，不显示技术异常。
- 登录页不展示基础档案或路线评估细节。

### 4.2 基础档案

默认结构：

- 运动频率。
- 单次运动时长。
- 户外经验。
- 近期爬升经验。
- 身高、体重、常用背包重量。

成功标准：

- 每个输入都说明用途。
- 可跳过。
- 不把档案包装成“能力成绩单”。

### 4.3 首页

默认结构：

- 顶部 TrailMate、天气位置、通知。
- 标题“准备走哪条线？”
- 导入 GPX 主入口。
- 当前路线卡。
- 三个快速入口：路线评估、现场导航、装备检查。
- 没有真实数据时不显示今日步数、热量等假概览。

### 4.4 路线 Tab

默认结构：

- 当前路线卡。
- 导入/更换 GPX。
- 目标路线列表。
- 导入状态只在进行中或失败时出现。

路线 Tab 不显示 GPS 控制、全屏导航面板、高德诊断或完整评估结论。

### 4.5 路线详情

路线详情保留四个 Tab：

- 评估：路线是否适合。
- 路线：普通地图预览、路线结构、检查点。
- 计划：时间线、补给点、休息点、风险点。
- 装备：本路线装备建议和匹配。

全屏导航不再被当作路线 Tab 内的一块内容，而是从路线 Tab 进入的独立现场模式。

### 4.6 全屏导航

默认结构：

- 全屏地图。
- 返回按钮和路线名。
- 当前位置、路线折线、检查点、已记录轨迹。
- 当前检查点卡。
- 下一检查点和进度。
- GPS、记录、底图、装备状态条。
- 开始、暂停、继续、结束记录、安全分享。

不允许直接展示：

- 高德上线检查明细。
- 离线底图长篇说明。
- AI 装备建议。
- 基础档案或历史证据。

### 4.7 装备

路线装备清单：

- AI 建议横幅。
- 已匹配、需检查、缺失统计。
- 每项显示类别、理由、匹配装备或缺失状态。
- 缺失项主动作是“查看候选”或“选择候选”。

装备目录抽屉：

- 搜索框。
- 类别筛选。
- 服务端返回品牌、型号、重量和适用标签。
- 查看候选详情和缩略图。
- 找不到时提示更换关键词或等待服务端补充目录，不在手机端直接创建品牌装备。

不再单独提供“我的装备”作为本轮主流程。服务端装备目录是品牌、型号、图片和标签的可信来源；手机端只展示与当前路线相关的匹配候选。

## 5. 服务端边界

### 5.1 新增装备目录

服务端新增只读目录：

- `GET /api/v1/gear/catalog/categories`
- `GET /api/v1/gear/catalog/search?category=&q=`
- `GET /api/v1/gear/catalog/{catalogItemId}`

目录项字段：

- `catalogItemId`
- `category`
- `brand`
- `model`
- `displayName`
- `weightGrams`
- `tags`
- `source`

### 5.2 路线装备匹配

Android 使用目录项直接匹配本次路线装备需求：

- 服务端目录返回类别、品牌、型号、重量、标签和缩略图地址。
- Android 根据路线装备类别和类别别名寻找候选。
- UI 展示匹配结果与候选详情，不创建个人装备库存记录。

### 5.3 AI 装备建议

`POST /api/v1/plans/{planId}/gear-advice` 只返回路线需要的类别、状态、理由和可选匹配线索。它不直接创建用户装备，也不伪造品牌型号。

## 6. Android 实现边界

本轮实现应避免继续扩大 `RouteDetailScreen.kt` 和 `MyGearScreen.kt`：

- 将全屏导航状态和 UI 提取为独立现场模式组件。
- 将服务端装备目录 DTO、客户端和 UI 状态独立于本地 `GearInventory`。
- 将“添加已有装备”“我的装备”“保存到我的装备”替换为服务端目录匹配和候选详情。
- 不暴露手填品牌型号入口；装备选择必须来自服务端目录。

## 7. 真机验收

目标设备：SM_S9260，解锁密码 1027。

验收路径：

1. 安装 debug APK。
2. 进入登录页，确认微信登录入口和未配置/失败状态可读。
3. 完成或跳过基础档案。
4. 导入 GPX 并进入路线详情。
5. 进入装备 Tab，确认服务端目录候选、品牌型号和缩略图可见。
6. 打开候选详情，确认没有“我的装备”或手填品牌型号流程。
7. 进入全屏导航，确认没有诊断杂糅，GPS 与轨迹记录可用。

服务端验收：

1. 通过 `ssh remote-linux` 访问部署机器。
2. 确认 TrailMate 服务健康。
3. 确认装备目录 API 可返回种子数据。
4. Android 真机可请求该 API。

## 8. 非目标

- 不做商城、购买、价格、库存或导购。
- 不做社区路线库。
- 不做完整 turn-by-turn 导航。
- 不承诺救援或安全保证。
- 不做 iOS。

## 9. 自检

- 设计聚焦方向 A，没有扩大到商业化或社区。
- 装备主路径改为服务端目录选择。
- 全屏导航不再展示诊断和准备说明。
- 登录注册和基础档案保持私密输入定位。
- 真机和 remote-linux 验收标准明确。
