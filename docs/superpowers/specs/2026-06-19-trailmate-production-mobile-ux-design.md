# TrailMate 生产级手机端产品体验设计规格

日期：2026-06-19
状态：设计草案，待用户确认后进入逐页实现
范围：Android 手机端信息架构、页面职责、交互动线、视觉系统和实施顺序

## 1. 设计目标

TrailMate 手机端要从“功能原型”升级为生产级徒步 App。生产级不等于堆更多入口，而是让用户在每个阶段都能清楚知道下一步要做什么：

- 行前：导入路线，理解这条线是否适合自己，补齐装备和离线准备。
- 途中：用地图、GPS、检查点和轨迹记录完成轻导航。
- 事后：查看轨迹复盘，把真实活动沉淀为能力证据。

页面必须围绕行动组织，不展示 AI prompt、历史证据包、调试诊断或注册资料细节。用户档案、历史 GPX、身高体重、装备库存是评估输入，不是主界面内容。

## 2. 当前产品状态判断

当前 Android 端已经具备真实 Alpha 骨架：

- `HomeScreen.kt` 有首页、路线工作台、装备、数据、我的等入口。
- `RouteDetailScreen.kt` 已支持评估、路线、计划、装备 Tab，并接入地图、GPS、轨迹记录、偏航、安全分享、离线状态和高德诊断。
- `MyGearScreen.kt` 已支持路线清单、我的装备、装备详情和添加品牌装备。
- `OnboardingScreens.kt` 已采集账号、运动习惯、户外经验、身高、体重、背包重量。

主要问题：

- `HomeScreen.kt` 和 `RouteDetailScreen.kt` 文件过大，页面和编排职责混杂，后续设计迭代风险高。
- 首页、路线工作台、路线详情之间存在职责重叠，用户容易看到多个“评估/装备/离线/GPS”入口。
- 路线详情里的“现场详情”“高德上线检查”“地图图层解释”等信息虽然有用，但生产 UI 中必须降级为可展开详情，不能和地图驾驶舱争首屏。
- 我的页目前承担基础档案、历史活动、数据控制等多个方向，信息结构还不像真实账户/设置中心。

## 3. 产品定位

TrailMate 不是路线社区，也不是专业救援工具。它是：

> 基于个人能力证据的徒步行前评估、装备准备和现场轻导航工具。

差异点：

- 用用户自己的历史活动和基础档案做个性化路线评估。
- 把 AI 放在“解释、装备建议、补给休息建议”上，而不是让 AI 直接决定路线分数。
- 把装备从静态清单变成路线相关的准备状态。
- 用离线优先的地图、GPS 和轨迹记录支持真实户外场景。

## 4. 竞品启发

这些启发来自当前公开产品资料和项目已有研究文档：

- AllTrails 把离线地图、偏航提醒、路线条件放在“路线信心”能力中。
- komoot 把户外导航集中在路线规划、离线地图、语音/转向指引和无信号可用。
- Gaia GPS 的移动端核心是显示位置、切换图层、离线地图、记录轨迹、创建路线和同步数据。
- Strava Beacon 说明安全分享应该出现在记录场景里，而不是藏在普通设置。

TrailMate 的取舍：

- 不追求社区路线库和完整 turn-by-turn。
- 优先保证导入路线后的行前评估、装备准备、离线状态、真实 GPS 和记录轨迹。
- 安全分享属于路线驾驶舱和记录过程，不是“我的”页的普通设置。

## 5. 设计原则

### 5.1 页面只承担一个主任务

每个页面必须有一个主问题：

- 首页：今天要准备哪条路线？
- 路线：我有哪些路线，当前路线准备到哪一步？
- 路线详情：这条路线是否适合、怎么走、怎么准备？
- 装备：这次路线缺什么，我已有装备是否可用？
- 数据：我走过什么，能力是否变化？
- 我的：账号、隐私、权限和基础设置在哪里？

### 5.2 现场信息优先

途中页面必须优先显示地图、当前位置、下一检查点、记录状态和主动作。诊断、解释、依据和设置只能作为二级信息。

### 5.3 证据后台化

历史 GPX、基础档案、身高体重和装备详情可以影响结果，但不在首页或路线首屏裸展示。需要解释时，用“为什么这样建议”的摘要，而不是展示完整证据材料。

### 5.4 克制但有质感

视觉参考用户给出的三张图：象牙白背景、深苔绿主色、地图大画面、浅灰边线、清晰状态卡和 iOS 式轻盈交互。避免厚重暗色、过多渐变、过多卡片套卡片。

### 5.5 安全边界明确

所有建议保持辅助语气，不承诺安全、不替代离线地图和路标、不提供救援保证。

## 6. 信息架构

底部 Tab 保持五个，但职责需要收敛：

| Tab | 产品角色 | 首屏主任务 | 禁止放置 |
| --- | --- | --- | --- |
| 首页 | 今日行动入口 | 选择/导入路线，进入当前路线 | 长篇证据、全部数据报表 |
| 路线 | 路线库和路线工作台 | 管理目标路线，查看当前准备状态 | 个人资料编辑、历史能力细节 |
| 装备 | 个人装备和路线清单 | 补齐本路线关键装备 | 商城化推荐、无关品牌广告 |
| 数据 | 轨迹复盘和能力趋势 | 看已记录活动和趋势 | 导入/编辑基础档案主流程 |
| 我的 | 账号、权限、隐私 | 管理个人设置和数据权利 | 路线评估详情、装备清单主流程 |

路线详情内部保留四个 Tab：

| Tab | 产品角色 | 默认内容 |
| --- | --- | --- |
| 评估 | 行前判断 | 难度、时间、关键风险、可信度、下一步 |
| 路线 | 现场轻导航 | 地图驾驶舱、GPS、轨迹、检查点 |
| 计划 | 时间和补给休息 | 检查点、补给/休息建议、回撤提醒 |
| 装备 | 本路线装备 | AI 清单、已有装备匹配、补齐入口 |

## 7. 用户动线

### 7.1 首次使用

1. 用户看到注册/登录页。
2. 完成基础档案采集，明确可跳过。
3. 进入首页，主动作是“导入 GPX 路线”。
4. 首页只显示当前路线推荐卡和快速入口，不展示内部证据。

关键体验：基础档案不是结果页，而是 AI/评估输入。采集完成后不应让用户停留在“档案成绩单”。

### 7.2 行前准备

1. 用户导入目标路线。
2. App 显示解析状态和路线摘要。
3. 用户进入路线详情的“评估”Tab。
4. 评估卡给出“推荐 / 谨慎 / 暂不建议”、预计用时和 3 个以内关键风险。
5. 主动作引导用户去“装备检查”或“路线驾驶舱”。

关键体验：不要让用户先读一堆证据。先给判断，再给解释。

### 7.3 装备准备

1. 用户在路线详情或底部装备 Tab 进入装备清单。
2. 默认看到“本路线必备项”。
3. 已匹配装备显示品牌型号；缺失项提供“添加已有装备”。
4. 我的装备作为二级 Tab，用于维护库存。

关键体验：装备页不是购物页。品牌型号只来自用户已保存装备。

### 7.4 现场徒步

1. 用户进入路线详情的“路线”Tab。
2. 首屏是地图驾驶舱。
3. 状态条显示 GPS、记录、离线、装备。
4. 主按钮根据状态变化：启用定位、开始徒步、暂停、继续、查看恢复、查看回顾。
5. 偏航时停止自动推进检查点，优先显示恢复建议和安全分享。

关键体验：现场页面不能让用户翻很长页面找按钮。

### 7.5 事后复盘

1. 用户结束记录。
2. App 进入轨迹回顾。
3. 数据页展示距离、时长、爬升、轨迹点、与计划偏差。
4. 复盘结果沉淀为历史能力证据。

关键体验：数据页展示结论和趋势，不展示原始 AI 输入。

## 8. 页面设计规格

### 8.1 注册 / 登录

主任务：创建私密账户并进入基础档案。

首屏结构：

- TrailMate 标识。
- 一句安全边界文案。
- 邮箱、密码。
- 主按钮“继续”。
- 次按钮“已有账号登录”。
- 隐私和位置数据说明入口。

优化要求：

- 当前“注册 / 登录”和“已有账号，直接登录”可以保留，但登录/注册状态需要用分段控件或底部切换区分。
- 不在此页解释完整产品能力。
- 密码错误、邮箱格式、网络失败必须有就地反馈。

### 8.2 基础档案采集

主任务：快速采集路线评估所需的最低上下文。

首屏结构：

- 顶部进度：“基础档案 1/2”。
- 分组：平时运动、户外经验、身体/背包。
- 每组最多 3 个选择或输入。
- 主按钮“保存并进入首页”。
- 次按钮“暂时跳过”。

优化要求：

- 身高体重标为可选，并说明只用于负重/耗能保守估算。
- 不展示“能力证据”。
- 可后续在我的页编辑。

### 8.3 首页

主任务：让用户开始今天的路线准备。

首屏结构：

- 顶部：TrailMate、天气/位置、通知。
- 问候标题：“准备走哪条线？”
- 主入口：导入 GPX。
- 当前路线评估卡：路线图/山脊图、路线名、距离、爬升、预计用时、风险标签、下一步。
- 快速开始：路线评估、轻导航、装备清单。
- 今日概览：步数/热量类数据当前可以删除或降级，除非有真实数据来源。

优化要求：

- 今日概览如果没有真实传感器或 Health Connect 数据，应改为“最近记录”或隐藏。
- 快速开始最多 3 个入口。
- 当前路线卡必须是首页视觉主角。
- 历史 GPX 和基础档案不在首页直接展开。

### 8.4 路线 Tab / 路线工作台

主任务：管理目标路线并进入当前路线详情。

首屏结构：

- 当前路线卡。
- 导入路线入口。
- 路线列表或导入队列。
- 空状态：导入 GPX 或使用示例路线。

优化要求：

- 不重复展示完整路线评估；评估属于路线详情。
- 导入队列只在有任务时出现。
- 历史活动导入不放在路线工作台首屏；它属于数据/我的中的能力证据管理。

### 8.5 路线详情 - 评估

主任务：回答“这条线对我是否合适”。

首屏结构：

- 路线名、距离、爬升。
- 分段控件：评估 / 路线 / 计划 / 装备。
- 评估结果卡：推荐、谨慎、暂不建议。
- 预计用时区间。
- 3 个以内关键风险。
- 主动作：去路线驾驶舱或检查装备。

优化要求：

- 证据展开为“查看依据”，默认折叠。
- 可信度用简短标签，不展示长证据列表。
- 不出现技术诊断。

### 8.6 路线详情 - 路线驾驶舱

主任务：现场轻导航。

首屏结构：

- 地图占 58%-65% 高度。
- 地图上显示参考路线、检查点、当前位置、已记录轨迹。
- 浮层显示当前检查点、下一检查点、进度。
- 下方动作抽屉。
- 状态条：GPS、记录、离线、装备。

优化要求：

- “现场详情”“地图图层”“高德上线检查”折叠到二级区。
- 状态条必须可点击并进入对应工作流。
- 地图失败时显示可恢复空状态，但保留主动作。

### 8.7 路线详情 - 计划

主任务：告诉用户何时补给、休息、核对风险。

首屏结构：

- 计划摘要：预计完成、检查点数量、关键风险。
- 检查点时间线。
- 每个检查点显示距离、预计时间、动作建议。

优化要求：

- 文案用“建议检查”，不用“必须补水”。
- 同类检查点合并，不超过 12 项。
- 支持离线查看。

### 8.8 路线详情 - 装备

主任务：检查本路线装备是否就绪。

首屏结构：

- AI 装备建议横幅。
- 清单进度：已匹配 / 需检查 / 缺失。
- 必备装备列表。
- 缺失项入口“添加已有装备”。

优化要求：

- 路线详情里的装备只显示本路线清单，不展示完整库存管理。
- 完整库存管理跳到底部“装备”Tab。
- 品牌型号只显示用户已有装备。

### 8.9 装备 Tab

主任务：维护个人装备库并匹配当前路线。

结构：

- 顶部路线选择胶囊。
- 三个内部 Tab：路线清单 / 我的装备 / 详情。
- 路线清单优先。
- 我的装备用于库存管理。
- 详情用于单件装备或匹配详情。

优化要求：

- 添加装备使用底部抽屉，不打断当前清单。
- 装备状态用图标和颜色，不用大段说明。
- “Beta”标签可保留，但不要像核心卖点。

### 8.10 数据 Tab

主任务：复盘实际活动和能力趋势。

首屏结构：

- 最近轨迹复盘卡。
- 计划 vs 实际摘要。
- 历史活动列表。
- 能力趋势入口。

优化要求：

- 如果没有轨迹，展示“完成一次记录后会出现复盘”。
- 历史 GPX 作为能力证据管理入口，不在首页展示。
- 导出 GPX 属于单次记录详情或更多菜单。

### 8.11 我的 Tab

主任务：账户、隐私、权限和设置。

结构：

- 个人基础档案摘要。
- 权限状态：定位、通知、地图授权。
- 数据与隐私：导出、清除本地数据、删除账户。
- 地图设置：高德授权、离线地图入口。
- 关于与安全边界。

优化要求：

- 不展示路线评估和装备清单主流程。
- 基础档案只显示摘要和编辑入口。
- 历史活动证据只显示“已导入 N 条，可管理”。

## 9. 视觉系统

### 9.1 色彩

保留当前 `TrailMateTheme.kt` 的方向：

- 主色：`TrailMateMoss` 深苔绿，用于主按钮、选中态、已就绪状态。
- 强调色：`TrailMateClay` 和 `TrailMateAmber`，用于谨慎、缺失、警告。
- 背景：`TrailMateMist` 象牙白。
- 卡片背景：白色和 `TrailMateField`。
- GPS/路线色：蓝色仅用于当前位置、轨迹进度、地图状态。

调整要求：

- 不使用大面积单色深绿背景。
- 风险色只出现在风险、缺失、提醒状态。
- 保持浅色基调，暗色模式后续单独校准。

### 9.2 布局

- 页面水平边距：20-24dp。
- 卡片圆角：8-16dp，地图浮层可 16-20dp。
- 底部导航固定。
- 地图、路线卡、装备清单是视觉主角；小状态不应抢视觉。
- 避免页面级大卡片套小卡片。

### 9.3 字体层级

- 页面标题：28-34sp。
- 卡片标题：18-22sp。
- 状态值：16-20sp。
- 说明文字：12-14sp。
- 不用 viewport 缩放字体。

### 9.4 图标

- 底部 Tab 使用线性图标。
- 状态条用图标表达：GPS、记录、离线、装备。
- 按钮优先图标 + 短文案。
- 不用文本当图标。

### 9.5 动效与触感

生产实现阶段应加入：

- Tab 切换的轻量淡入。
- 状态条点击的底部抽屉或展开详情。
- 地图浮层状态变化的平滑过渡。
- 开始/暂停/继续记录按钮的明确状态反馈。

## 10. 组件边界建议

后续实现不应继续扩大 `HomeScreen.kt` 和 `RouteDetailScreen.kt`。建议逐步拆分：

- `feature/home/HomeDashboardScreen.kt`
- `feature/route/RouteWorkspaceScreen.kt`
- `feature/route/detail/RouteDetailScreen.kt`
- `feature/route/detail/RouteAssessmentTab.kt`
- `feature/route/detail/RouteCockpitTab.kt`
- `feature/route/detail/RoutePlanTab.kt`
- `feature/route/detail/RouteGearTab.kt`
- `feature/data/DataScreen.kt`
- `feature/profile/ProfileSettingsScreen.kt`
- `core/design/TrailMateCards.kt`
- `core/design/TrailMateBottomSheets.kt`

拆分目标是让每个文件有单一职责，不改变现有业务能力。

## 11. 推荐推进路径

### 方案 A：先视觉统一，再拆信息架构

优点：短期截图好看。
缺点：功能职责仍混乱，后面会反复返工。

### 方案 B：先信息架构和页面职责，再逐页视觉精修

优点：动线稳定，后续改造不容易堆功能。
缺点：第一轮看起来可能只是“收敛”，不是立刻大变样。

### 方案 C：直接重写整套页面

优点：视觉可以一次性焕新。
缺点：风险高，现有 GPS、GPX、装备、持久化和测试容易被破坏。

推荐：方案 B。先把页面职责和入口收敛，再逐页做视觉精修。每一页都用现有测试保护，不做大爆炸重写。

## 12. 第一阶段实施范围

第一阶段只做手机端信息架构和视觉基础，不新增服务端：

1. 收敛首页：去掉无真实数据支撑的今日运动概览，强化当前路线卡和导入入口。
2. 拆出路线工作台：路线 Tab 只管理路线，不重复完整评估。
3. 精简路线详情首屏：评估和路线驾驶舱各自只显示核心任务。
4. 装备页去购物化：保留路线清单和已有装备匹配。
5. 数据页改成复盘中心：历史 GPX 和轨迹记录合并为活动/能力输入。
6. 我的页改成设置中心：档案、权限、隐私、地图设置。

## 13. 验收标准

设计实现后，每一页必须满足：

- 首屏能用一句话说明当前主任务。
- 首屏最多一个主按钮。
- 没有 AI prompt、证据包、调试诊断直接出现在主视觉区。
- 底部 Tab 间没有职责重复。
- 现场路线页在不滚动时可看到地图、当前位置/检查点、主动作和状态条。
- 装备缺失、GPS 未授权、离线未准备等关键状态都有明确入口。
- 空状态给出可执行动作。
- 所有安全相关文案保持辅助语气。

## 14. 非目标

本设计规格不要求本轮实现：

- 完整服务端账号系统。
- 社区路线库。
- 商城、广告或品牌推荐。
- 完整 turn-by-turn 导航。
- 自动救援或安全承诺。
- KML 解析和导入链路，除非后续单独实现解析、测试和导入验证。
- iOS 版本。

## 15. 规格自检

- 无未定义页面：已覆盖注册、基础档案、首页、路线、路线详情四 Tab、装备、数据、我的。
- 无证据泄漏：所有历史 GPX、身体数据和 AI 输入均定义为后台输入或折叠解释。
- 无职责重复：每个底部 Tab 和路线详情 Tab 都有唯一主任务。
- 实施可渐进：推荐方案 B，保留现有业务能力和测试。
- 安全边界明确：不承诺救援、不替代地图和现场判断。
