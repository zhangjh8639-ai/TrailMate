# TrailMate 全 App 交互与动线审计

日期：2026-06-20

范围：Android Compose 原型、现有截图证据、路线/GPS/离线地图 QA 文档。

## 市场参照

成熟户外和运动 App 的共同模式不是“把能力都展示出来”，而是按场景切换密度：

- AllTrails：路线详情服务行前判断，导航/记录是进入路线后的现场模式。
- Komoot：先规划路线，再强调离线可用和导航过程。
- Gaia GPS：地图、当前位置、离线地图、轨迹记录是核心工具，尤其重视出发前下载。
- Strava：记录时把安全分享放在现场动作附近，而不是藏到个人设置。

TrailMate 的差异化应保留在“个人能力 + 路线风险 + 装备准备”的 AI 评估，但这些证据不应在前台堆叠。前台只呈现结论、原因摘要和下一步。

## 推荐主动线

1. 首次使用：注册/本地档案 -> 基础体能与户外经验 -> 地图服务授权 -> 定位权限说明。
2. 首页：只回答今天要走哪条线、是否已准备好，以及继续动作。
3. 路线：导入或更换 GPX，确认目标路线存在。
4. 路线详情/评估：判断这条线是否适合我，给出风险和主动作。
5. 装备：按路线缺口补齐装备，并把用户品牌装备绑定到清单。
6. 计划：查看补给、休息、风险检查点。
7. 现场导航：全屏地图 + 当前检查点 + GPS/记录/离线/安全分享。
8. 数据：结束后复盘轨迹、查看历史活动，历史证据只做后台评估输入。
9. 我的：个人资料、权限、隐私、地图服务、离线地图管理、数据导出/清除。

## 页面职责审计

### 首页

职责：给用户一个最短入口。

当前状态：视觉方向接近参考图，导入路线、当前路线、快速开始、今日概览结构清楚。

需要收束：

- “轻导航”快捷入口在没有路线或未通过准备时容易造成跳步。建议改成状态化入口：无路线为“导入路线”，有路线为“继续准备”，进行中为“返回导航”。
- 今日概览在 MVP 可以保留，但不要抢路线准备的第一优先级。

### 路线 Tab

职责：路线文件与当前目标路线管理。

当前状态：无路线态清楚；有路线后可以进入详情。

需要收束：

- 路线 Tab 不应承载现场地图状态、GPS 开关、轻导航解释或高德诊断。
- 有路线后首屏建议展示“四项准备摘要”：评估、装备、离线底图、GPS/记录权限。点击后进入对应页或设置。

### 路线详情/评估

职责：回答“这条线适不适合我”。

当前状态：默认进评估，符合用户之前选择的 D 方向。

需要收束：

- 评估页只展示结论、关键指标、最多 3 个风险、可信度、下一步动作。
- 用户画像、历史证据、AI 输入、身高体重等后台证据不在主页面展示。

### 路线详情/路线

职责：行前查看路线结构和进入现场模式。

当前问题：从截图看，路线 Tab 仍像“地图预览 + 轻导航 + 状态诊断”的混合体。地图画布、当前检查点、开始徒步、定位状态、底图状态都在同一首屏，用户会不清楚这是规划页还是行进页。

建议改成两层：

- 行前路线页：地图预览、路线提示点、检查点列表入口、一个主动作“进入导航”。
- 全屏导航页：真正显示 GPS、记录、进度、安全分享、标记检查点、结束记录。

“地图状态”不应作为路线 Tab 独立模块保留。它应变成“地图与离线设置”折叠项，只在离线底图未下载、授权缺失、SDK 异常或用户主动检查时出现。

### 计划 Tab

职责：补给、休息、风险点时间线。

当前方向：作为独立 Tab 合理。

需要收束：

- 文案应是建议语气，不做安全承诺。
- 点位详情应支持“设为关注点”，但不要把每个点都做成复杂卡片堆叠。

### 装备 Tab

职责：当前路线的装备清单 + 我的装备匹配。

当前状态：已经接近用户给的第三张参考图方向。

需要收束：

- “添加已有装备”应优先从缺失项进入底部弹层，减少全局表单常驻。
- 品牌/型号/可用状态是用户资产信息，路线清单只展示匹配结果和修复动作。

### 数据 Tab

职责：轨迹复盘与历史活动管理。

当前状态：已经移除了首屏 AI 证据口吻，方向正确。

需要收束：

- 历史 GPX 是后台能力证据，不应成为主导航里的高频任务。
- 结束徒步后从全屏导航进入数据复盘，比用户主动点数据 Tab 更自然。

### 我的

职责：账户、资料、权限、隐私、地图服务。

需要补强：

- 高德 Key/包名/SHA1 这类工程诊断不应在普通路线页出现。
- 面向用户的设置应是“地图服务状态”“离线地图管理”“定位权限”“数据与隐私”。
- 开发诊断可以放在开发者折叠区或 debug build 专属入口。

## 当前最高优先级

1. 把路线 Tab 的行前地图和全屏导航完全拆开。路线 Tab 不直接承载“开始徒步”。
2. 把“地图状态/高德上线检查”降级为问题态或折叠设置，默认首屏不展示。
3. 首页快捷入口改为单一主线，不让“轻导航”在未准备时跳过评估和装备。
4. 装备添加改为从缺失项进入底部弹层。
5. 继续证明真实离线底图下载和真机 GPS 记录，不把 emulator 结果写成生产完成。

## 地图与户外生产边界

路线地图状态需要分四层表达，不能合并成一个“可用”标签：

- 在线底图可用：高德 Key、SDK、隐私同意都就绪，MapView 可以加载在线瓦片。
- 本地路线包可用：GPX 几何、检查点和路线数据已保存在本机，可用于无网查看路线骨架。
- 目标区域离线底图可用：用户已在高德离线地图管理中下载实际徒步区域，SDK saved-region 状态可验证，并且已确认该区域覆盖当前 GPX 路线。
- 户外生产可用：真机完成 GPS 精度、锁屏记录、弱信号、通知控制、安全分享、耗电和离线地图现场验证。

因此路线页可以显示“在线可用”，但不能把在线底图和本地路线包直接写成“生产可用”。这与 AllTrails、komoot、Gaia GPS、OsmAnd 的常见模式一致：行前必须把离线地图和定位可靠性当作明确准备项，现场导航再显示轨迹记录和安全动作。

## 本轮实现证据

- 普通路线 cockpit 不再直接暴露“开始徒步”、暂停/继续或标记检查点；现场动作通过“进入导航/全屏导航”进入专门模式。
- 全屏导航保留“开始徒步”、安全分享、标记点、结束记录，并新增下一检查点动作提示，例如补水、补能量或休息判断。
- 换路线时保留用户所在 Tab，但按新路线重置徒步 session、GPS 和轨迹状态，避免旧路线现场状态污染新路线。
- 地图 readiness 文案已从“生产可用”降级为“在线可用”，并提示出发前确认目标区域离线底图，避免把在线瓦片误表达为户外生产闭环。
- AMap 在线地图表面与户外生产 readiness 布尔已拆开：在线底图可渲染，但不会把 `isProductionMapReady` 置为 true。
- 出发检查已将“本地路线包”和“目标区域离线底图”拆成独立项；未检测到覆盖当前路线的离线底图区域时，主动作会指向“下载目标区域离线底图”，例如“下载杭州市离线底图”，不能直接显示“可以出发”。
- 离线底图覆盖判断已从“下载区域数量”推进到“目标路线反地理编码区域 + 已下载城市/省份元数据”匹配；这仍不等于断网瓦片已现场验证。
- 高德上线诊断也已接入同一覆盖判断：仅有离线区域数量时，不会显示“可真机验证”。
- 路线 cockpit 主按钮现在跟随出发检查修复顺序：未开始徒步前，会先引导保存路线包、下载目标区域离线底图、飞行模式验证底图、授权定位或补齐装备，不会提前显示“开始徒步”。
- 路线 cockpit 开始动作已加入可靠定位门禁：GPS 搜索中、缺少精度或精度弱时，不直接显示“开始徒步”。
- 展开后的地图准备信息已从“地图状态”改为“地图与离线”，默认路线首屏继续隐藏工程化诊断。
- 安全分享现在要求可靠定位精度：缺少精度或精度差于 100 m 时，不生成坐标链接，只引导用户重新定位。
- 生产默认路线导入页不展示“使用示例 GPX”；只有测试/demo 调用显式打开时才保留示例路线入口。
- 新增户外生产发布门禁：模拟器地图/记录回归通过仍会显示“不可发布”，必须补齐 release 身份、目标区域离线底图、真机现场 QA、后台记录、弱信号、耗电和安全分享证据后，才可进入“生产候选”。
- 发布门禁已进一步收紧：仅有离线区域数量不够，必须证明覆盖当前路线，并完成断网瓦片验证。
- 已通过 5 个 focused Android Compose 回归测试：全屏导航启动/推进、路线页 GPS/记录控件收束、换路线重置现场状态、偏离路线恢复入口、首页进入全屏导航隐藏底部栏。

仍未完成生产闭环：

- 高德离线底图区域下载仍未成功完成，不能宣称真实离线底图能力完成。
- 真机户外 QA 仍未执行，不能从模拟器结果推断 GPS 精度、锁屏记录、弱信号、耗电和安全分享在真实徒步中可靠。
