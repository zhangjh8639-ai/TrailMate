# TrailMate 路线详情四 Tab 重设计分析

日期：2026-06-20
状态：设计输入，下一步进入实现规格

## 目标

路线详情页需要从“功能证明页”升级为“真实徒步产品页”。四个 tab 的分工应足够清晰：

1. 评估：出发前回答“这条线适不适合我”。
2. 路线：行进中回答“我在哪、下一步去哪、现在按哪个按钮”。
3. 计划：出发前回答“何时补给、休息、核对风险”。
4. 装备：出发前回答“还缺什么、已有装备能不能带”。

本次重点问题是“路线 / 轻导航”tab：当前它已经有地图、GPS、轨迹记录、高德状态和诊断能力，但信息层级像工程面板。用户会看到太多 SDK、授权、图层、GPS、通知、离线包和 readiness 细节，导致主任务被冲淡。

## 外部产品模式

### AllTrails

AllTrails 的路线体验强调路线选择、离线地图、GPS 导航、记录活动和偏航提醒。对 TrailMate 的启发：

- 路线详情和现场导航要分层：详情页帮助判断是否要走，导航页帮助用户现场行动。
- 偏航、GPS 误差、离线问题属于现场状态提示，不应做成首屏诊断清单。
- 离线能力应在行前准备中主动提示，导航中只显示是否可用。

参考：

- https://www.alltrails.com/welcome
- https://support.alltrails.com/hc/en-us/articles/360019246391-Resolving-GPS-errors-on-Android

### komoot

komoot 强调路线规划、离线地图、户外导航和语音指引。对 TrailMate 的启发：

- 计划和导航是两个场景：计划页安排时间、补给、休息；导航页只保留行进决策。
- 行前准备可以使用 checklist，但导航中不应让 checklist 抢地图空间。
- 用户现场需要大按钮和明确动作，而不是复杂设置。

参考：https://www.komoot.com/features

### Gaia GPS

Gaia GPS 的核心是地图、当前位置、离线地图和轨迹记录。对 TrailMate 的启发：

- 地图是导航页主体，轨迹记录是底层能力，不是单独堆叠的大模块。
- 图层、导出、GPS 精度、记录详情可以放在二级 sheet 或设置页。
- 户外场景应降低视觉噪声，保证弱网、强光和行进中可扫读。

参考：https://help.gaiagps.com/hc/en-us/articles/9067661557399-How-to-Use-Gaia-GPS

### Strava Beacon

Strava Beacon 把安全分享放在活动记录场景中。对 TrailMate 的启发：

- 安全分享是行进中的辅助动作，适合放在轻导航控制台，不适合放在个人设置或装备页。
- 分享内容应基于当前路线、当前位置和记录状态，不宣称救援能力。

参考：https://support.strava.com/hc/en-us/articles/224357527-Strava-Beacon

## 平台与 SDK 授权原则

### 高德 SDK 隐私合规

高德 Android SDK 的隐私合规要求应在初始化或创建 SDK 组件前完成。TrailMate 当前已经把 MapView 进入条件放在本地 consent 之后，但同意入口藏在路线 tab 的展开诊断里，体验不自然。

推荐：

- 首次打开 App 或首次进入地图能力前，展示“地图与定位服务说明”。
- 用户勾选同意后，保存 `AmapPrivacyConsent`。
- 未同意时不初始化高德 MapView，只显示本地 GPX 预览。
- 轻导航页不展示“高德授权”大卡，除非用户主动打开“地图设置”或当前地图无法使用。

参考：

- https://lbs.amap.com/news/sdkhgsy
- https://lbs.amap.com/api/android-sdk/guide/create-project/android-studio-create-project

### Android 定位与通知权限

Android 官方建议在用户理解功能价值的上下文中请求运行时权限。TrailMate 不应在用户第一次打开 App 时连续弹出定位和通知系统弹窗；这会像权限轰炸，也容易被拒绝。

推荐：

- App 首次使用时展示产品内说明页，说明定位、通知、地图 SDK 的用途。
- 用户点击“启用 GPS”或“开始徒步记录”时，请求前台定位权限。
- 用户点击“开始记录轨迹”且 Android 版本需要通知权限时，请求通知权限。
- “我的 / 权限状态”只做查看与修复入口，不承担首次授权主流程。

参考：

- https://developer.android.com/training/permissions/requesting
- https://developer.android.com/develop/ui/views/notifications/notification-permission

## 当前实现审计

### 评估 Tab

当前优点：

- 已经从地图复用改成结论优先。
- 首屏有结论、置信度、核心风险和下一步动作。
- 没有展示历史证据和用户身体数据。

当前问题：

- 视觉仍偏“系统卡片”，缺少路线详情页的仪式感和层级。
- “检查装备 / 进入路线”两个按钮并列，但没有根据评估结论改变主次。
- 风险解释是纯文本列表，缺少可扫读的风险分类：天气、暴露、距离、爬升、装备。

重设计方向：

- 首卡改成“路线适配结论”：颜色、图标、短文案更像参考图里的黄色谨慎卡。
- 根据评估结果动态主动作：
  - 推荐：主按钮“进入路线”，次按钮“检查装备”。
  - 谨慎：主按钮“检查装备”，次按钮“查看计划”。
  - 不建议：主按钮“调整路线/查看替代建议”，次按钮“仍要查看路线”。
- 风险展示从句子列表改成 2 到 4 个风险 chip：天气、暴露、长距离、信号弱、装备缺口。
- “为什么这样判断”折叠到二级，不在首屏展开 AI/历史证据。

### 路线 Tab / 轻导航

当前优点：

- 已经有地图主体、当前位置按钮、当前检查点、进度、主动作、安全分享、标记点。
- 已经支持高德地图和本地 GPX 预览双模式。
- 已经有真实 GPS、轨迹记录、离线包、安全分享等能力。

当前问题：

- 高德授权、离线包、定位精度、图层说明、SDK 诊断都聚集在“地图与定位设置”下，像工程调试入口。
- 同一屏同时出现“现场状态”“地图与定位设置”“readiness strip”“action drawer”，状态重复。
- Action drawer 是卡片叠在地图底部，但内容较多，按钮数量偏多，现场操作容易分心。
- “启用 GPS”“开始徒步”“开始记录”之间的关系还不够清晰：用户会疑惑到底是开启定位，还是开始导航，还是开始记录轨迹。
- 地图 readiness 过多使用“高德接入准备中”“Key 待配置”“SDK 待接入”这类开发语言，不应默认暴露给真实用户。

重设计方向：

路线 tab 应成为“导航 cockpit”，只保留三层：

1. 地图层：GPX、检查点、当前位置、精度圈、已记录轨迹。
2. 导航控制台：当前检查点、下一检查点、进度、一个主按钮。
3. 快捷工具条：安全分享、标记点、离线/地图源、结束或更多。

推荐首屏结构：

- 地图接近全屏，顶部只放返回、路线名短标题、地图源状态。
- 地图左侧保留定位按钮，右侧可保留图层/更多按钮。
- 底部控制台采用半高 bottom sheet：
  - 状态行：在线路 / GPS 弱 / 偏离路线 / 等待定位。
  - 当前：起点、CP2、已完成距离。
  - 下一站：名称、距离、预计用时或爬升。
  - 主按钮：启用 GPS / 开始徒步 / 暂停 / 继续 / 查看恢复建议 / 查看轨迹。
  - 辅助操作：安全分享、标记点、结束。
- 高德状态只显示为用户语言：
  - “在线底图可用”
  - “当前使用离线 GPX”
  - “地图服务需同意后启用”
  - “地图不可用，继续使用本地路线”
- SDK key、package、SHA1、MapView、注册 activity 等仅放入开发者诊断或内部 debug 页面，不进入普通用户路线详情。

状态机建议：

| 状态 | 用户看到 | 主动作 | 二级入口 |
| --- | --- | --- | --- |
| 未同意地图服务 | 当前使用本地路线 | 继续本地路线 / 同意地图服务 | 地图设置 |
| 未授权定位 | 等待 GPS | 启用 GPS | 手动查看路线 |
| 定位中 | 正在定位 | 继续等待 | 地图设置 |
| 已定位未开始 | 起点准备 | 开始徒步 | 安全分享、离线 |
| 记录中 | 当前检查点 | 暂停 | 标记点、安全分享、结束 |
| GPS 弱 | GPS 较弱 | 继续记录 | 查看定位建议 |
| 偏离路线 | 请核对方向 | 查看恢复建议 | 安全分享、标记点 |
| 已完成 | 已完成路线 | 查看轨迹回顾 | 保存/分享 |

### 计划 Tab

当前优点：

- 已改为检查点时间线。
- 行前准备四件事：离线地图、补给休息、装备清单、天气，方向正确。
- 不再展示地图或技术诊断。

当前问题：

- “补给与休息计划”首卡偏概要，缺少实际行程节奏。
- 检查点卡的信息密度还不够像路线计划：缺少预计到达时间、补给建议、海拔/爬升段提示。
- “天气稳定”如果没有真实天气来源，容易让用户误解为确定判断。

重设计方向：

- 首卡改为“今日行程计划”：总时长、出发建议、最晚折返点。
- 时间线每项展示：
  - 到达里程
  - 预计用时区间
  - 行动建议：补水、补能量、休息、天气复核、下撤判断
  - 风险 badge
- 天气没有真实 API 时文案改成“天气待确认”或“请出发前复核”，不使用“稳定”这种确定性词。
- 增加“离线保存计划”状态，但不重复路线 tab 的地图 readiness。

### 装备 Tab

当前优点：

- 已支持 AI 装备建议、本地 fallback、我的装备匹配。
- 缺失装备可以跳转添加。

当前问题：

- 当前 `RouteGearTab` 主要是 `TrailMatePanel` 纵向列表，视觉不如用户参考图里的装备清单精致。
- 缺少“路线清单 / 我的装备 / 详情”的明确二级 tab。
- 装备项没有强弱优先级和图像/图标层级，用户扫读成本高。
- “AI 装备建议”字样适合 Beta 提醒，但不应成为列表主角；用户要的是“这次要带什么”。

重设计方向：

- 首屏顶部：路线名 + 适配结论 + “路线清单 / 我的装备 / 详情”分段。
- AI 建议用一条轻提示，不抢主标题。
- 清单按重要性排序：
  - 必带
  - 建议检查
  - 可选
  - 已匹配
- 装备项结构参考用户截图：
  - 左侧状态圆点：已匹配 / 需检查 / 缺失。
  - 中部图标或装备图。
  - 标题：雨衣、防水头灯、登山杖。
  - 副文案：已匹配品牌型号 / 电量建议 / 缺失原因。
  - 右侧动作：查看匹配 / 添加已有装备。
- 添加装备使用底部 sheet，不离开上下文。

## 新的信息架构

### 首次使用 / 授权准备

新增“路线能力准备”轻量流程，触发时机：

- 用户首次完成 onboarding 后。
- 或用户首次进入路线详情并尝试使用地图/GPS/记录。

内容：

1. 地图服务说明：高德 SDK 用于在线底图，未同意也可使用本地 GPX。
2. 定位说明：用于当前位置、路线匹配和轨迹记录。
3. 通知说明：用于记录中切后台的持续状态。
4. 用户选择：
   - 同意地图服务并继续
   - 稍后，在本地路线中继续

系统权限弹窗只在用户触发具体能力时出现：

- 启用 GPS 时请求定位。
- 开始轨迹记录时请求通知。

### 四 Tab 关系

1. 评估：决策页，主动作引导去装备/计划/路线。
2. 路线：地图 cockpit，只服务现场行进。
3. 计划：行前节奏页，补给、休息、天气复核。
4. 装备：路线装备清单与个人装备匹配。

四个 tab 共享路线名、距离、爬升，但不要共享同一套大卡片。

## 视觉方向

以用户参考图为主：

- 背景：温暖白、浅雾绿、淡地形纹理。
- 主色：深苔绿，作为主按钮和已准备状态。
- 行进色：蓝色，专用于 GPS、当前位置和路线进度。
- 风险色：琥珀和克制红，仅用于谨慎、偏航、缺失。
- 卡片：少嵌套，半径 16 到 24，关键操作区域可用更大圆角。
- 地图：不要放在小卡片里，路线 tab 地图应是第一视觉。
- 文案：短句、动作优先、避免“SDK / Key / package / SHA1 / diagnostics”等工程词。

## 后续实现优先级

1. 新增首次地图/定位准备 flow，把高德隐私同意从路线 tab 展开项前移。
2. 重构路线 tab 首屏：地图 + 单一导航控制台 + 工具条，移除默认诊断区。
3. 将高德 SDK 诊断迁移到开发者/设置入口，普通用户只看地图服务状态。
4. 重设计装备 tab 为真实清单样式。
5. 微调评估和计划 tab 的视觉和文案层级。
